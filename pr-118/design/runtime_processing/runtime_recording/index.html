    Runtime recording#

    CARETはアプリケーションの実行中でも計測が可能です。

    これにより、実行中の必要な時間の評価が行いやすいようになっています。 本セッションでは、途中からの計測機能の詳細を説明します。

    See also:

    • Software architecture | caret_trace

    背景#

    CARETは Tracepointで説明している通り、はランタイムではトレースポイントを極力減らすため、メタ情報は初期化時に保存するようになっています。

    そのため、実行時のオーバーヘッドはひくいのですが、 単にLTtngセッションを途中から　起動するだけでは、計測結果が何ノードのコールバックか特定できません。 A LTTng session has to be started before running the application.

    ところが、実運用では途中から計測死体ケースは多くあります。

    概要#

    CARETは途中からの記録を可能にするために、初期化時の情報をメモリに格納しておき、計測開始後にトレースデータに保存させています。

    TraceNodeは以下の４つの状態を持つ。

    • 待機状態(WAIT)
    • 計測準備状態(PREPARE)
    • 計測状態(RECORD)

    今回の提案では、以下のようにlttng sessionを起動した後に「記録開始トピック」をpublishして、実際に記録を開始させる。

    利用方法の互換性を保つために、事前にsessionを起動するケースでは、記録開始トピックのpublishもせずに記録を開始できるようにしておく。

    Topic#

    topic_name 役割
    /caret/start_record 計測の開始
    /caret/end_record 計測の終了
    /caret/status 計測状態の同期

    Start.msg#

    uint32 recording_frequency 100
string ignore_nodes  # reserved
string ignore_topics # reserved
string select_nodes  # reserved
string select_topics # reserved

    Status.msg#

    string caret_node_name
int8 status
string[] node_names # reserved
int64 pid # reserved

    End.msg#

    (Empty)

    Info

    他の方法として、初期化関連のトレースポイントはblockモードで記録して、 計測関連のトレースポイントはdiscardモードで記録すれば、初期化関連のトレースポイントは確実に保存される。 ロストの発生自体はこれで抑えられる可能性はあるが、今回はcaret側で改善を行うこととした。

    TraceNodeの状態#

    主な状態遷移を示す。

    各状態での全イベント（start_recording/end_recording）に対する状態遷移を記述したものを以下に示す。

    WAIT#

    静的なトレースポイントの保存のみを行う。 「/caret/start_record」トピックの受信で「PREPARE」へ遷移

    PREPARE#

    メモリ上に保存した静的なトレースポイントの記録のみを行う。 この際のトレースポイント記録の周波数は任意に設定できること。 計測は行わない。

    RECORD#

    計測中の状態。 全トレースポイントが同期的に記録される。 静的なトレースポイントはメモリ上に保存する。
